Analyzing the Impact of CI Sub-practices on Continuous Code Quality in Open-Source Projects: An Empirical Study


PAPER INFORMATIONS


Guilherme Freitas
UFRN, Brazil


João Helis Bernardo
IFRN, UFRN, Brazil


Gustavo Sizílio
IFRN, Brazil


Daniel Alencar da Costa
University of Otago, New Zealand


Uirá Kulesza
UFRN, Brazil


In this work, we conduct an exploratory study investigating how the adoption of a CI service (Travis CI) impacts on the Continuous Code Quality (CCQ) inspection outcomes of 75 relevant open-source projects hosted on GitHub for a period of 2 years. We further complemented our analysis by investigating to what extent four CI sub-practices, including avoiding long-running builds, fixing broken builds quickly, controlling for acceptable test coverage, and integrating at least daily, impacts on the CCQ outcomes of 97 open-source projects from GitHub.

In particular, we address the following research questions:

RQ1. What is the impact of adopting a CI service on the projects' quality inspection outcomes?

Surprisingly, our statistical analyses indicate that the adoption of Travis CI shows no significant association with improved CCQ inspection outcomes. This finding emphasizes that the mere adoption of a CI service does not guarantee automatic improvements in quality outcomes for open-source projects. Further investigations are needed to explore the underlying factors that influence the relationship between CI adoption and quality outcomes.

Violin plots for the comparisons performed in RQ1



RQ2. To what extent the adoption of CI sub-practices share a relationship with improved quality inspection outcomes?

By analyzing four CI sub-practices, including avoiding long-running builds, fixing broken builds quickly, controlling for acceptable test coverage, and integrating at least daily, our results indicate that only controlling for test coverage and avoiding long-running builds have a significant association with improved quality inspection outcomes. These findings underscore the importance of prioritizing test coverage and minimizing build duration as key factors in enhancing software quality throughout the development process.

Datasets

For replication purpose, we publicize our datasets to the interested reader.



In the dataset of RQ1, we provide the values of the metrics bugs, code smells, duplicated lines, and technical debts for a total of 144 versions of 75 relevant open-source projects.

The metrics were collected from the SonarQube platform, which is a widely used platform for continuous inspection of code quality.

* The first 48 versions represent the versions before the adoption of Travis CI.
* The next 48 versions (from 49 to 96) represent the versions of the first year after the adoption of Travis CI ("early CI").
* The last 48 versions (from 97 to 144) represent the versions of the third year after the adoption of Travis CI ("late CI").

The table below detail the meta-data that we fetch for all 144 versions of the 75 projects investigated in RQ1.

# Variable Definition
1 Version The software version number for which the metric was calculated.
2 Metric The specific metric calculated for the software version. This dataset presents data for five metrics (bugs, code_smells, duplicated_lines, code_smells, sqale_debt_ratio, and nloc).
3 Date The date when the metric was collected in SonarQube.
4 Value The value of the metric for the specific software version.
5 normalized_value The value of the metric normalized by the nloc.
6 project_name The name of the project for which the metric was calculated.

The dataset of RQ2 is composed of data of 97 open-source projects from GitHub. For each project, we provide the values for metrics related to CI sub-practices, such as Build Duration (sec.), Time to Fix broken builds, Test Coverage (%), and CI build activity. Furthermore, our dataset also has the values of the metrics related to continuous code quality, such as bugs, code smells, duplicated lines, and technical debts for such projects.

The metrics Build Duration (sec.), Time to Fix broken builds, and CI build activity were collected from Travis CI API, while the Test Coverage (%), bugs, code smells, duplicated lines, and technical debts were collected from SonarQube API.
